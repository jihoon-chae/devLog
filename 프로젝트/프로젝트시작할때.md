### 1. 컨벤션

🙋 “컨벤션을 어떻게 정하고 어떻게 시작해야 할까요?”

- 컨벤션을 정하려는 이유는? 가독성이나 나중에 코드를 정리할 때 불편할 테니 원활한 협업을 위해!
- 팀원끼리 조율하여 정하기. 오히려 다 정하고 시작하려는 것이 진입장벽이 될 수 있음
- 간단하게 [Prettier](https://prettier.io/) 설정으로 맞출 수 있는 것을 정하고 가는 것도 좋은 방법
- 컨벤션도 프로젝트 진행하며 논의하며 지속해서 리팩토링

### 2. 프로젝트 진행 방식(feat. 애자일 방법론)

🙋 “프로젝트 경험이 처음이라 어떻게 진행하면 좋을까요?”
🙋 ”애자일 방법론으로 프로젝트를 진행하려면 어떡해야 할까요?”

- 진행 방식은 팀의 스타일이기 때문에 정해진 답은 없음. 팀원들끼리 소통하여 결정하는 것이 바람직
- To-do list, Notion, GitHub Projects 등 도구는 편한 것으로 사용
- 단 효율적인 협업을 위해 사용하는 것이기에 도구를 익히는 것 자체가 허들이 된다면 그 또한 효율적이지 않을 수 있음
- 보통 회사에서는 기획-디자인-개발로 돌아가니 애자일 방법론을 많이 사용하지만, 이번 팀 프로젝트 같은 경우에는 기능이 정해져 있어 맞지 않을 수 있음
- 너무 방법론적인 접근보다는 때로는 우리 팀의 스타일대로 진행하다 맞는 방법을 찾을 수도 있음

### 3. 프로젝트를 진행하며 가장 중요한 것

🙋 “프로젝트에서 제일 중요한 것은 어떤 것일까요?”
🙋 “어떤 기능을 사용해봤다나 협업을 해봤다 일까요?”

- 질문에 원초적으로 접근해서 정답을 찾아보자. “팀 프로젝트를 왜 해야할까?”와 “팀 프로젝트에서 무엇을 얻을 수 있을까?”를 생각해 보자.
- 프로젝트 시작 전에 미리 내가 이 프로젝트를 통해 이뤄낼 것을 정리하고 들어가도 좋은 방법. 기능 구현에 몰입하다 보면 잊는 경우도 있으니 꾸준히 상기 필요
    - 기능적인 부분이라면 리액트를 써보겠어부터 Sass를 능숙하게 사용해보겠어, 키보드 접근성을 맞춰보겠어, 리액트의 이 Hook은 꼭 제대로 써보겠어 등의 구체적인 목표를 세우는 것이 좋음
    - 협업적인 부분이라면 회사에서 발생할 수 있는 것들을 미리 해보겠다거나 커뮤니케이션에서 발생하는 문제를 어떻게 해결해나가겠다 등
- 이러한 목표와 진행 상황을 꾸준히 문서화하여 나중에 리드미에 녹여내는 것도 좋음
- 팀 프로젝트지만 포트폴리오로 면접을 보는 것은 본인이기 때문에 본인이 이 팀 안에서 어떤 역할을 했는지도 꾸준히 정리
    - 꼭 개인으로 특출나게 잘해야 한다는 것이 아니라 본인이 노력한 부분을 적으면 충분히 어필이 될 것

### 4. 그 외

**1) API 명세 작성 방법**

- 잘 적는 것은 타인이 이해를 잘 할 수 있게 명확하게, 누가 읽어도 이해할 수 있도록 적는 것
- REST API에 대해 공부하면 도움이 될 것

**2) 마지막 한두 마디**

- 프로젝트를 진행하며 원하던 방향대로 가지 않을 수 있지만 끝까지 포기하지 말고 유종의 미를 거둘 수 있는 경험이 되었으면 좋겠다. 최종 완성의 여부와는 별개로 내가 경험하고, 포폴에 어필할 수 있는 점이 분명히 있다고 생각한다.
- 예를 들어 면접 질문으로 팀 내 의견 충돌을 어떻게 해결했는지에 경험을 녹일 수 있다.
- 프로젝트의 기술적 난이도도 중요하지만, 결과적인 완성보다는 우리 팀이 이런 문제를 어떻게 고민하고 해결했다는 일련의 과정을 적는 것도 도움이 된다 생각한다. 꾸준히 기록하자.

## 커밋 메시지 양식

|태그|설명|
|---|----|
|Feat:|새로운 기능을 추가할 경우|
|Fix:|버그를 고친 경우|
|Design:|CSS 등 사용자 UI 디자인 변경|
|BREAKING CHANGE:|커다란 API 변경의 경우|
|HOTFIX:|급하게 치명적인 버그를 고쳐야하는 경우|
|Style:|코드 포맷 변경, 세미 콜론 누락, 코드 수정이 없는 경우|
|Refactor:|프로덕션 코드 리팩토링|
|Comment:|필요한 주석 추가 및 변경|
|Docs:|문서를 수정한 경우|
|Test:|테스트 추가, 테스트 리팩토링(프로덕션 코드 변경 X)|
|Chore:|빌드 태스트 업데이트, 패키지 매니저를 설정하는 경우(프로덕션 코드 변경 X)|
|Rename:|파일 혹은 폴더명을 수정하거나 옮기는 작업만인 경우|
|Remove:|파일을 삭제하는 작업만 수행한 경우|

1. URL 설계

4. URL에 따라서 컴포넌트가 제대로 나오는지
5. 1개 페이지에 1개 컴포넌트로 일단 만들기
6. 컴포넌트를 세분화 (공책에 간략아게 연결되는 다이어그램 그리기)
